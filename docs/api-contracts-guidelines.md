# API Contracts Guidelines

## Source of Truth

All protobuf definitions originate from the **Buf Schema Registry** module `buf.build/project-kessel/inventory-api`. This repository does **not** contain `.proto` files. Generated Java code is committed directly to the repository under `kessel-sdk/src/main/java`.

## Code Generation Workflow

### Rules

1. **Never hand-edit files under these paths** -- they are auto-generated and marked `DO NOT EDIT`:
   - `org.project_kessel.api.inventory.v1*` (all versioned packages)
   - `com.google.api.*`, `com.google.rpc.*`
   - `build.buf.validate.*`
2. Run `buf generate` to regenerate all stubs. Configuration lives in `buf.gen.yaml` at the repo root.
3. Generated output lands in `kessel-sdk/src/main/java` via two remote plugins:
   - `buf.build/protocolbuffers/java` -- message classes, enums, OrBuilder interfaces
   - `buf.build/grpc/java` -- `*Grpc.java` service stubs (blocking, async, future)
4. A GitHub Actions workflow (`buf-generate.yml`) runs `buf generate` every 6 hours and opens a PR automatically on branch `buf-generate-update`. Do not manually push generated code to `main`; merge the automated PR instead.
5. When releasing, run `buf generate` locally before `mvn clean install` only if proto updates are needed between automated runs.

### Identifying Generated vs. Hand-Written Code

| Pattern | Generated? |
|---|---|
| `@com.google.protobuf.Generated` annotation | Yes |
| `@io.grpc.stub.annotations.GrpcGenerated` annotation | Yes |
| `// Generated by the protocol buffer compiler.` header | Yes |
| Files in `org.project_kessel.api.inventory` root (e.g., `AbstractClientBuilder.java`) | **No** -- hand-written |
| Files in `org.project_kessel.api.auth`, `org.project_kessel.api.grpc`, `org.project_kessel.api.rbac.v2` | **No** -- hand-written |
| `ClientBuilder.java` inside versioned packages | **No** -- hand-written SDK wrapper |

## API Versioning Strategy

### Version Hierarchy

| Package | Status | Purpose |
|---|---|---|
| `v1beta1` | Legacy | Resource-specific CRUD services (per-type: K8sCluster, K8sPolicy) and separate authz service |
| `v1beta2` | **Current / active** | Unified `KesselInventoryService` with generic resource lifecycle + authz checks |
| `v1` | Health only | `KesselInventoryHealthService` (livez/readyz probes) |

### Rules

1. **Target `v1beta2`** for all new consumer code. The `v1beta1` packages exist only for backward compatibility and contain per-resource-type services that have been superseded.
2. Version appears in the gRPC service name (`kessel.inventory.v1beta2.KesselInventoryService`), the Java package (`org.project_kessel.api.inventory.v1beta2`), and the HTTP transcoding path (`/api/kessel/v1beta2/*`).
3. Each version is fully self-contained: message types (e.g., `CheckRequest`, `ResourceReference`) are defined per-version, not shared across versions.
4. The `v1beta1` sub-package structure uses domain scoping: `authz/` for check operations, `resources/` for resource CRUD, `relationships/` for relationship CRUD. The `v1beta2` model flattens all operations into a single `KesselInventoryService`.

## Service Contracts

### v1beta2 KesselInventoryService -- Complete RPC Catalog

| RPC | Method Type | Request | Response |
|---|---|---|---|
| `Check` | Unary | `CheckRequest` | `CheckResponse` |
| `CheckSelf` | Unary | `CheckSelfRequest` | `CheckSelfResponse` |
| `CheckForUpdate` | Unary | `CheckForUpdateRequest` | `CheckForUpdateResponse` |
| `CheckBulk` | Unary | `CheckBulkRequest` | `CheckBulkResponse` |
| `CheckSelfBulk` | Unary | `CheckSelfBulkRequest` | `CheckSelfBulkResponse` |
| `CheckForUpdateBulk` | Unary | `CheckForUpdateBulkRequest` | `CheckForUpdateBulkResponse` |
| `ReportResource` | Unary | `ReportResourceRequest` | `ReportResourceResponse` |
| `DeleteResource` | Unary | `DeleteResourceRequest` | `DeleteResourceResponse` |
| `StreamedListObjects` | **Server streaming** | `StreamedListObjectsRequest` | `StreamedListObjectsResponse` (stream) |
| `StreamedListSubjects` | **Server streaming** | `StreamedListSubjectsRequest` | `StreamedListSubjectsResponse` (stream) |

### Rules

1. **Check vs. CheckSelf**: `Check` requires an explicit `SubjectReference`. `CheckSelf` derives the subject from the caller's authentication context and omits the subject field.
2. **Check vs. CheckForUpdate**: `Check` tests read-style permissions. `CheckForUpdate` tests write/edit-style permissions. Both share the same structure (`object`, `relation`, `subject`).
3. **Bulk variants** accept a repeated list of `*BulkRequestItem` messages and return `*BulkResponsePair` entries. Each pair contains the original request echoed back alongside either an `item` (success) or an `error` (`google.rpc.Status`). Always check `hasItem()` / `hasError()` on each pair.
4. No client-streaming or bidirectional-streaming RPCs exist. Streaming is **server-side only** for list operations.

## Request/Response Patterns

### Core Message Types (v1beta2)

- **`ResourceReference`**: `resource_type` + `resource_id` + optional `ReporterReference` (reporter `type`).
- **`SubjectReference`**: wraps a `ResourceReference` with an optional `relation`.
- **`RepresentationType`**: `resource_type` + `reporter_type` -- used in streaming list requests to identify object/subject types.
- **`ReporterReference`**: identifies the reporting system by `type` string.

### Consistency Model

Read requests (`Check*`, `StreamedList*`) accept an optional `Consistency` field with exactly one of:
- `minimize_latency` (bool, must be `true`) -- fastest snapshot available
- `at_least_as_fresh` (`ConsistencyToken`) -- snapshot at least as fresh as a known token
- `at_least_as_acknowledged` (bool, must be `true`) -- includes all acknowledged writes

If omitted, the server defaults to `minimize_latency` (deployments may override to `at_least_as_acknowledged`).

### Write Visibility

`ReportResourceRequest` includes a `write_visibility` enum:
- `WRITE_VISIBILITY_UNSPECIFIED` / `MINIMIZE_LATENCY` -- fire-and-forget, lower latency
- `IMMEDIATE` -- waits for acknowledgment; enables subsequent reads with `at_least_as_acknowledged`

### Pagination (Streaming Lists)

- **Request**: `RequestPagination` with `limit` (uint32, required) and optional `continuation_token`.
- **Response**: `ResponsePagination` with `continuation_token`. An empty string signals the final page.
- Iterate by reading `continuation_token` from each `StreamedListObjectsResponse.pagination` and passing it on the next request. See `ListWorkspaces.java` for the canonical pagination iterator pattern.

### Allowed Enum

`CheckResponse.allowed` returns:
- `ALLOWED_UNSPECIFIED` (0) -- unset/unknown
- `ALLOWED_TRUE` (1) -- access granted
- `ALLOWED_FALSE` (2) -- access denied

Always compare against enum constants, never raw integers.

## SDK Client Wrappers

### ClientBuilder Pattern

Each versioned service has a hand-written `ClientBuilder` extending `AbstractClientBuilder<BlockingStub, AsyncStub>`:

```java
// Blocking client
Pair<KesselInventoryServiceBlockingStub, ManagedChannel> pair =
    new ClientBuilder(endpoint).insecure().build();

// Async client
Pair<KesselInventoryServiceStub, ManagedChannel> pair =
    new ClientBuilder(endpoint).insecure().buildAsync();
```

### Rules

1. `build()` returns a `Pair<BlockingStub, ManagedChannel>`. Always call `channel.shutdown()` in a finally block.
2. Authentication modes: `.insecure()`, `.oauth2ClientAuthenticated(creds)`, `.authenticated(callCreds, channelCreds)`, `.unauthenticated()`.
3. Combining `CallCredentials` with `InsecureChannelCredentials` throws `IllegalStateException` at build time -- this is enforced by `validateCredentials()`.
4. For OAuth2, use `OIDCDiscovery.fetchOIDCDiscovery(issuerUrl)` to resolve the token endpoint, then pass `OAuth2ClientCredentials` to the builder.

## RBAC Utility Layer

The `org.project_kessel.api.rbac.v2` package provides hand-written helpers that build on top of `v1beta2` protobuf types:

- `Utils` -- factory methods for common `ResourceReference`, `SubjectReference`, `RepresentationType` builders (workspace, role, principal).
- `FetchWorkspace` -- HTTP client for fetching workspace metadata from the RBAC REST API (not gRPC).
- `ListWorkspaces` -- pagination-aware iterator over `StreamedListObjects` for workspace listing.
- `Workspace` -- Jackson-annotated POJO for workspace JSON responses.

## Backward Compatibility Rules

1. **Never remove or renumber existing fields** in proto definitions (controlled upstream at `buf.build/project-kessel/inventory-api`).
2. **Enum zero values** are always `*_UNSPECIFIED`. Client code must handle unrecognized enum values (the `UNRECOGNIZED(-1)` sentinel).
3. **`buf.validate` annotations** enforce required fields at the proto level. SDK consumers must satisfy these (e.g., `object`, `relation`, `subject` in `CheckRequest` are all required via `buf.validate.field`).
4. The surefire plugin excludes generated proto classes from test execution: all `v1` and `v1beta1` packages, plus specific patterns in `v1beta2` (`*Proto.java`, `*OrBuilder.java`, `*OuterClass.java`, `KesselInventoryServiceGrpc.java`). Hand-written tests live alongside the SDK wrapper code in `org.project_kessel.api.auth`, `org.project_kessel.api.inventory`, and `org.project_kessel.api.rbac.v2`.

## Dependency Versions

Managed in the parent `pom.xml`:
- `grpc.version` -- gRPC Core, Protobuf, Stub (runtime); Testing, Netty-shaded (test-only)
- `protobuf.version` -- Protobuf Java runtime
- `nimbus-oauth.version` -- OAuth2/OIDC SDK (marked `optional`)
- `jackson.version` -- JSON processing for RBAC REST helpers
- Java target: **21**

When bumping gRPC or Protobuf versions, always regenerate stubs via `buf generate` and verify with `./mvnw clean verify`.
