# Inventory Package Guidelines

Rules for `org.project_kessel.api.inventory` and its versioned sub-packages -- the `AbstractClientBuilder` hierarchy, `ClientBuilder` subclasses, and the generated/hand-written code boundary.

## Generated vs Hand-Written Boundary

This is the most important rule in this package.

| Path | Status |
|------|--------|
| `AbstractClientBuilder.java` (this directory) | **Hand-written** -- the core builder |
| `v1/ClientBuilder.java` | **Hand-written** -- health service builder |
| `v1beta2/ClientBuilder.java` | **Hand-written** -- inventory service builder |
| Everything else in `v1/`, `v1beta1/`, `v1beta2/` | **Generated** -- do not edit |

- Generated files carry `@com.google.protobuf.Generated`, `@io.grpc.stub.annotations.GrpcGenerated`, or a `// Generated by the protocol buffer compiler.` header.
- A GitHub Actions workflow regenerates stubs every 6 hours and opens a PR on branch `buf-generate-update`. Manual edits to generated files will be overwritten.
- `v1beta1` is legacy -- it has generated stubs but no hand-written `ClientBuilder`. Do not extend it.

## AbstractClientBuilder Pattern

- `AbstractClientBuilder<BT extends AbstractStub<BT>, AT extends AbstractAsyncStub<AT>>` is parameterized on blocking and async stub types.
- All security logic (TLS defaults, credential validation, channel construction) lives here. Never duplicate security logic in subclasses.
- Concrete `ClientBuilder` subclasses are trivial (~9 lines) and only implement `newStub(Channel)` and `newAsyncStub(Channel)`.

### Adding a New Service ClientBuilder

1. Create `ClientBuilder` in the new versioned package.
2. Extend `AbstractClientBuilder<NewBlockingStub, NewAsyncStub>`.
3. Implement `newStub(Channel)` and `newAsyncStub(Channel)` -- call `*Grpc.newBlockingStub(channel)` and `*Grpc.newStub(channel)`.
4. Use static imports for the stub inner classes: `import static ...ServiceGrpc.ServiceBlockingStub;`
5. Do not add any security, TLS, or credential logic to the subclass.

## Channel Security Rules

- Default channel credentials: `TlsChannelCredentials.create()` when none are explicitly set. Do not change this default.
- `insecure()` sets `InsecureChannelCredentials.create()` and forces `callCredentials` to null. Never modify it to preserve `CallCredentials`.
- `validateCredentials()` throws `IllegalStateException` if `CallCredentials` are combined with insecure channel credentials. Never remove or weaken this guard.
- `isChannelCredentialsSecure()` recursively checks `CompositeChannelCredentials` and `ChoiceChannelCredentials`. If gRPC introduces new credential types, update this method.

## Pair Return Convention

- `build()` returns `Pair<BlockingStub, ManagedChannel>` using `com.nimbusds.jose.util.Pair` (not Apache Commons).
- `buildAsync()` returns `Pair<AsyncStub, ManagedChannel>`.
- The caller owns the `ManagedChannel` and must shut it down. Every example demonstrates `channel.shutdown()` in a `finally` block (blocking) or in both `onCompleted()`/`onError()` (async).
- Each `build()`/`buildAsync()` call creates a new `ManagedChannel`. Do not call `build()` per-request -- reuse the channel.

## Builder Thread Safety

- `AbstractClientBuilder` is **not** thread-safe during construction (mutable builder pattern).
- Build it on a single thread, then share the resulting stub and channel freely -- gRPC stubs and channels are thread-safe.

## Authentication Modes

| Method | Channel | Call Credentials |
|--------|---------|-----------------|
| `oauth2ClientAuthenticated(creds)` | TLS (default) | OAuth2 Bearer |
| `oauth2ClientAuthenticated(creds, chanCreds)` | Explicit | OAuth2 Bearer |
| `authenticated(callCreds)` | TLS (default) | Custom |
| `authenticated(callCreds, chanCreds)` | Explicit | Custom |
| `unauthenticated()` | TLS (default) | None |
| `unauthenticated(chanCreds)` | Explicit | None |
| `insecure()` | Plaintext | None (forced) |

## Version Targeting

- **Use `v1beta2`** for all new feature work. It provides the unified `KesselInventoryService`.
- **`v1`** is health-check only (`GetLivez`, `GetReadyz`).
- **`v1beta1`** is legacy. Do not add a `ClientBuilder` or extend it.

## Transport Agnosticism

- The SDK does not bundle a gRPC transport. `grpc-netty-shaded` is test-scope only.
- Consumers must add their own transport runtime (e.g., `grpc-netty-shaded` or `grpc-okhttp`).
- Never add a runtime dependency on a specific gRPC transport.

## Testing

- Tests for `AbstractClientBuilder` live in `kessel-sdk/src/test/java/org/project_kessel/api/inventory/AbstractClientBuilderTest.java`.
- Use a concrete inner `TestClientBuilder` subclass to test the abstract class.
- Surefire excludes all `v1` and `v1beta1` packages and specific generated file patterns in `v1beta2`. Do not place hand-written tests in excluded paths.
- Always shut down `ManagedChannel` in test cleanup after calling `build()` or `buildAsync()`.
