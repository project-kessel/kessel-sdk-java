## Repository Structure

```
kessel-sdk-java/                  # Parent POM (kessel-sdk-parent)
  kessel-sdk/                     # Published SDK artifact (org.project-kessel:kessel-sdk)
    src/main/java/
      org/project_kessel/api/
        auth/                     # Hand-written: OAuth2, OIDC, token management
        grpc/                     # Hand-written: gRPC call credentials adapter
        inventory/                # Hand-written: AbstractClientBuilder
          v1/                     # Hand-written ClientBuilder + GENERATED health service stubs
          v1beta1/                # GENERATED ONLY -- legacy, do not extend
          v1beta2/                # Hand-written ClientBuilder + GENERATED inventory service stubs
        rbac/v2/                  # Hand-written: RBAC utilities (FetchWorkspace, ListWorkspaces, Utils)
      com/google/api/             # GENERATED -- Google API annotations
      com/google/rpc/             # GENERATED -- Google RPC status
      build/buf/validate/         # GENERATED -- buf validation
    src/test/java/                # Mirrors src/main package structure
  examples/                       # Runnable examples (not published)
  docs/                           # Guideline files (indexed above)
```

## Generated vs Hand-Written Code Boundary

This is the single most important architectural distinction in this repo.

**Generated code** is produced by `buf generate` from `buf.build/project-kessel/inventory-api`. It is committed to the repo but must never be hand-edited. Generated files carry `@com.google.protobuf.Generated` or `@io.grpc.stub.annotations.GrpcGenerated` annotations, or a `// Generated by the protocol buffer compiler.` header. They live in:
- `org.project_kessel.api.inventory.v1` (all files except `ClientBuilder.java`)
- `org.project_kessel.api.inventory.v1beta1` (entire package)
- `org.project_kessel.api.inventory.v1beta2` (all files except `ClientBuilder.java`)
- `com.google.api`, `com.google.rpc`, `build.buf.validate`

**Hand-written code** is everything else: `auth/`, `grpc/`, `rbac/v2/`, `AbstractClientBuilder.java`, and the `ClientBuilder.java` files inside versioned packages. When in doubt, check for the `@Generated` annotation.

A GitHub Actions workflow (`buf-generate.yml`) regenerates stubs every 6 hours and opens a PR on branch `buf-generate-update`. Do not manually push generated code to `main`.

## Build and CI

- **Java 21** required (set via `maven.compiler.release` in parent POM).
- **Maven wrapper**: always use `./mvnw`, not a system `mvn`. Wrapper pins Maven 3.9.6.
- **CI pipeline** (`.github/workflows/build.yml`): runs `./mvnw clean verify` on JDK 21 / Ubuntu on every push and PR to `main`.
- **Dependabot** updates Maven and GitHub Actions dependencies daily.
- **Verify command**: `./mvnw clean verify` is the single command that builds, tests, and validates. Use it before any PR.
- **Multi-module build**: the parent POM uses `<modules>` to build `kessel-sdk` and `examples` together. The examples module is never published (`maven.deploy.skip=true`, `central.skip=true`).

## Naming Conventions

### Packages
- Group ID: `org.project-kessel` (hyphen) in Maven; `org.project_kessel` (underscore) in Java packages. This mismatch is intentional and standard for Maven.
- Hand-written SDK code: `org.project_kessel.api.{auth,grpc,inventory,rbac.v2}`
- Examples: `org.project_kessel.examples` with utilities in `org.project_kessel.examples.util`
- API versions appear as package segments: `v1`, `v1beta1`, `v1beta2`

### Classes
- Immutable data carriers use Java **records**: `ClientConfigAuth`, `RefreshTokenResponse`, `OIDCDiscoveryMetadata`. Records validate all fields via `Objects.requireNonNull` in their compact constructor.
- The `Workspace` class is a Jackson-annotated **POJO** (not a record) because it needs mutable setters for deserialization. Setters are package-private.
- Static utility classes (`OIDCDiscovery`, `Utils`, `FetchWorkspace`, `OAuth2CallCredentials`) have a private constructor or only static methods. Do not instantiate them.
- Builder classes are named `ClientBuilder` in each versioned package, all extending `AbstractClientBuilder`.
- Exception: `OAuth2Exception` -- the single custom exception type. Extends `RuntimeException`.

### Methods
- Builder methods return `this` for fluent chaining.
- Factory methods use descriptive names: `oauth2CallCredentials()`, `fetchRootWorkspace()`, `workspaceResource()`.
- Test methods: `test<What>` in camelCase (e.g., `testConstructorWithNullTarget`).

## Code Style

- **No code formatter enforced** in CI. Follow the existing style: 4-space indentation, opening brace on same line, standard Java import ordering.
- **Wildcard imports** are used for closely related types (e.g., `import com.nimbusds.oauth2.sdk.*;`, `import io.grpc.*;`, `import org.project_kessel.api.inventory.v1beta2.*;`). Follow this pattern for generated protobuf packages. Use explicit imports for unrelated packages.
- **Static imports** for gRPC stub inner classes: `import static ...KesselInventoryServiceGrpc.KesselInventoryServiceBlockingStub;`
- **Static imports** in tests: `import static org.junit.jupiter.api.Assertions.*;` and `import static org.mockito.Mockito.*;`
- **Javadoc** is minimal -- only the `AuthRequest` interface has doc comments. The `maven-javadoc-plugin` runs with `doclint=none` and `failOnError=false`. Do not add verbose Javadoc unless the method's contract is non-obvious.
- **No Lombok, no Spring, no dependency injection frameworks.** Everything is wired manually.

## Dependency Management

- All dependency versions are centralized as properties in the **parent POM** (`grpc.version`, `protobuf.version`, `nimbus-oauth.version`, `jackson.version`). Never declare versions in child POMs.
- **Nimbus OAuth SDK** (`com.nimbusds:oauth2-oidc-sdk`) is `<optional>true</optional>` in `kessel-sdk/pom.xml`. Any code using Nimbus must include a `Class.forName()` runtime guard.
- **gRPC transport** (`grpc-netty-shaded`) is test-scope only in the SDK. The SDK is transport-agnostic. Examples use `grpc-okhttp`.
- **`com.nimbusds.jose.util.Pair`** is used throughout the SDK (not Apache Commons or a custom Pair). This comes from the Nimbus JOSE dependency which is transitively available. `build()` and `buildAsync()` return `Pair<Stub, ManagedChannel>`.
- **Jackson** (`jackson-databind`) is a runtime dependency for the RBAC REST helpers (`FetchWorkspace`, `Workspace`).
- When bumping gRPC or Protobuf versions, always regenerate stubs via `buf generate` and run `./mvnw clean verify`.

## Architectural Patterns

### Abstract Builder Pattern
`AbstractClientBuilder<BT, AT>` is the single extension point for new services. Concrete `ClientBuilder` subclasses are trivial (~9 lines) and only implement `newStub(Channel)` and `newAsyncStub(Channel)`. All security logic (TLS defaults, credential validation, channel construction) lives in the abstract parent. Never duplicate security logic in subclasses.

### Optional Dependency Guard
Code that depends on optional libraries (currently only Nimbus) must call `Class.forName()` in the constructor or entry point and throw a descriptive exception naming the required Maven artifact. This pattern exists in `OAuth2ClientCredentials` and `OIDCDiscovery`.

### Pair Return Convention
`build()` returns `Pair<BlockingStub, ManagedChannel>` and `buildAsync()` returns `Pair<AsyncStub, ManagedChannel>`. The caller is responsible for shutting down the channel. Every example demonstrates `channel.shutdown()` in a `finally` block (blocking) or in both `onCompleted()`/`onError()` (async).

## Version Targeting

- **Use `v1beta2` for all new feature work.** It provides the unified `KesselInventoryService` with all CRUD and authz operations.
- `v1` is health-check only (`GetLivez`, `GetReadyz`).
- `v1beta1` is legacy. It has generated stubs but no hand-written `ClientBuilder` and should not be extended.

## PR and Workflow Expectations

- CI must pass (`./mvnw clean verify`) before merge.
- Do not hand-edit generated protobuf files. If proto changes are needed, they happen upstream at `buf.build/project-kessel/inventory-api` and arrive via the automated `buf-generate` workflow.
- New hand-written classes need corresponding tests mirroring the source package path. Test class name: `<ClassName>Test`. Test visibility: package-private (no `public`).
- Commits follow the `chore: ...` / imperative style seen in recent history.

## Common Pitfalls

1. **Editing generated code.** Files under `v1beta1` and most of `v1beta2` are regenerated every 6 hours. Manual edits will be overwritten. Check for `@Generated` annotations.
2. **Forgetting channel shutdown.** Every `build()`/`buildAsync()` call creates a new `ManagedChannel`. Failing to shut it down leaks threads and file descriptors.
3. **Sending auth over insecure channels.** The builder throws `IllegalStateException` at build time if `CallCredentials` are combined with `InsecureChannelCredentials`. Do not try to bypass this.
4. **Adding Nimbus as a required dependency.** It must stay `<optional>true</optional>`. Consumers who do not use OAuth should not be forced to pull it.
5. **Placing tests in excluded packages.** Surefire excludes `v1`, `v1beta1`, and generated patterns in `v1beta2`. Hand-written tests in those paths will be silently skipped.
6. **Using `com.google.rpc.Status` as `io.grpc.Status`.** Bulk response errors use the protobuf `com.google.rpc.Status` message, not the gRPC `io.grpc.Status` enum. They are distinct types.
7. **Declaring dependency versions in child POMs.** All versions are properties in the parent POM. Duplicating them breaks centralized management.
